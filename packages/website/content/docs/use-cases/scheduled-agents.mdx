## Overview

Many agents support scheduled tasks like checking email, syncing data, running reports, or posting updates. The agent is responsible for defining and running these tasks. ClawRun is responsible for making sure the sandbox is awake when they need to run.

## How it works

Scheduling in ClawRun is a collaboration between the agent and the platform.

**The agent owns the schedule.** It defines cron jobs in its own configuration (e.g. ZeroClaw uses its built-in cron system). ClawRun never creates or modifies scheduled tasks. It only reads them.

**ClawRun owns the lifecycle.** While the sandbox is running, ClawRun periodically asks the agent for its cron schedule. It stores the next upcoming run time in the state store (Redis). When the sandbox is sleeping, a periodic heartbeat checks whether a cron job is due soon and wakes the sandbox in time.

### Cron discovery

While the sandbox is running, ClawRun calls the agent's cron listing command (e.g. `zeroclaw cron list`) every heartbeat cycle (60 seconds). The agent returns its scheduled jobs with their next run times. ClawRun persists the earliest upcoming time to the state store so it survives sleep/wake cycles.

If the agent has no cron jobs, nothing is stored and the sandbox will only wake on incoming messages or manual start.

### Wake timing

The heartbeat endpoint runs on a schedule configured during deploy. On each tick, it checks whether a cron job is due within the **lead time** (default 60 seconds). If so, it wakes the sandbox from its last snapshot, giving the agent time to boot before the job fires.

On paid Vercel plans, the heartbeat runs every minute for precise scheduling.

<Callout type="warn">
On Vercel's free (Hobby) plan, cron jobs are limited to once per day. This means the heartbeat that checks for upcoming agent tasks only runs once every 24 hours, so scheduled agents cannot wake on a precise schedule.

**Workarounds:**

- **Upgrade to a paid Vercel plan** for per-minute heartbeats.
- **Use an external cron service** (e.g. [cron-job.org](https://cron-job.org), [Upstash QStash](https://upstash.com/docs/qstash), or a GitHub Actions scheduled workflow) to call your heartbeat endpoint at the frequency you need:
  ```
  POST https://<your-deployment>/api/v1/heartbeat
  Authorization: Bearer <CLAWRUN_CRON_SECRET>
  ```

This is a Vercel platform limitation, not a ClawRun one. Other providers may not have this restriction.

</Callout>

### Keep-alive window

When the sandbox is already running and a cron job is approaching, ClawRun keeps the sandbox alive instead of snapshotting it. The **keep-alive window** (default 15 minutes) prevents unnecessary sleep/wake cycles when a job is imminent.

For example, if a cron job runs at 9:00 AM and the sandbox is still active at 8:50 AM, ClawRun will hold it awake through the job rather than snapshotting and immediately waking again.

## Lifecycle decisions

Every 60 seconds, the sidecar inside the sandbox sends a heartbeat to ClawRun. The server evaluates three reasons to keep the sandbox alive, in order:

1. **Grace period.** The sandbox was recently created. Always keep alive for the configured active duration.
2. **File activity.** The agent is actively modifying files in its workspace.
3. **Cron schedule.** A cron job is due within the keep-alive window.

If none of these apply, ClawRun snapshots the sandbox and puts it to sleep. Wake hooks are registered so messages and cron triggers can bring it back.

## Examples

### Daily summary

An agent configured to "send a daily summary every morning at 9 AM UTC":

- The agent translates that into a cron job in its own config.
- ClawRun discovers the schedule and stores the next run time.
- The sandbox sleeps after going idle.
- At 8:59 AM, the heartbeat detects the job is due within 60 seconds and wakes the sandbox.
- The agent boots from snapshot and runs the task at 9:00 AM.
- After the job completes and no extend reasons remain, ClawRun snapshots and sleeps.

### Frequent polling

An agent configured to "check the status API every 30 minutes":

- ClawRun keeps the sandbox alive if the next check is within 15 minutes.
- If the sandbox sleeps between checks, ClawRun wakes it before the next one.
- Requires a per-minute heartbeat (paid Vercel plan) for precise timing.
